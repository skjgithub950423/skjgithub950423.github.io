git-flow 是什么

众所周知，git是一个源码管理系统，那么就不可避免的涉及到多人协作问题
那么就应该有一个统一的工作流程，例如开发一个新功能的流程、已经发布的项目出现bug的修复流程等，如果没有一个统一的流程，每个人都按照自己的习惯来，那么就极有可能出现问题
所以在我的理解中，git-flow就是将我们平常工作流程标准化，只需要一条指令就可以帮我们完成一系列的操作

git-flow 使用

如果你已经在使用git进行代码管理，或者通过阅读参考资料中文章了解了git的工作流程，那么你应该知道，项目应该存在两个长期分支：

  • master
  • devolop

这两个分支会存活在整个项目的生命周期中

以及可能存在的三种短期分支：

  • 功能分支(future branch)
  • 补丁分支(hotfix branch)
  • 预发分支(release branch)

功能开发

假如我们要开发一个新功能cutImage，不使用git-flow的流程：

  1. 从develop迁出一个新的分支cutImage（也可以从master，前提是master与develop分支进度相同）

  2. 切换到cutImage分支进行开发

  3. 经过一段时间的开发功能大致完成，将分支合并回develop

  4. 删除cutImage分支

release管理

创建release

当你认为现在在 “develop” 分支的代码已经是一个成熟的 release 版本时，这意味着：
第一，它包括所有新的功能和必要的修复；
第二，它已经被彻底的测试过了。
如果上述两点都满足，那就是时候开始生成一个新的 release 了：

有了一个 release 分支，再完成针对 release 版本号的最后准备工作（如果项目里的某些文件需要记录版本号），并且进行最后的编辑。

完成release

  1. git-flow 会拉取远程仓库，以确保目前是最新的版本。
  2. release 的内容会被合并到 “master” 和 “develop” 两个分支中去，这样不仅产品代码为最新的版本，而且新的功能分支也将基于最新代码。
  3. 为便于识别和做历史参考，release 提交会被标记上这个 release 的名字（在我们的例子里是 “1.1.0”）。
  4. 清理操作，版本分支会被删除，并且回到 “develop”。

备注：以上所有操作都是基于本地仓库，完成后不要忘记把代码推送到远程仓库
